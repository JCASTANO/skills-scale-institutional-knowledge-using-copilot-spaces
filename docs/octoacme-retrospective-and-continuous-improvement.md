# OctoAcme — Retrospective & Continuous Improvement

## Purpose
Capture learnings and convert them into actionable improvements.

## When
After each sprint, release, or important milestone. Also after incidents.

## Structure
- What went well
- What could be improved
- Action items (owner assigned by **role**, due date, and linked issue/ticket)
- Follow-up on previous action items

## Running a Retrospective
- Timebox: 45–75 minutes depending on team size
- Facilitated by the **Scrum Master / Delivery Lead**
- Use an anonymous idea board if needed to encourage candor
- Prioritize 2–3 top action items to avoid overload
- Assign each action item to a specific named role (not just a team)

## Tracking Improvements
- Add action items to the project backlog as GitHub issues, with:
  - A clear owner identified by **role** (e.g., "QA Lead to improve regression coverage")
  - A due date or target sprint
  - Acceptance criteria so completion is verifiable
- Review outstanding actions in the weekly PM sync
- The **Scrum Master** is accountable for following up on all open retrospective actions
